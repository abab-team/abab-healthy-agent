# 开发规则

本文档定义 family-health-agent 的日常开发规则。所有开发必须按 `CODEX_IMPLEMENTATION_PLAN.md` 的 Phase 顺序推进，当前任务之外的后续 Phase 不得提前实现。

## 分层规则

### API 层

- 只负责接收请求、认证上下文提取、参数校验、调用 service 或 agent_service、响应映射。
- 不写业务决策。
- 不直接访问数据库。
- 不直接调用 LLM。
- 不绕过权限系统。

### Service 层

- 负责业务逻辑和业务规则。
- 可以调用 repository。
- 不依赖 LangGraph。
- 不直接调用真实 LLM SDK。
- 返回结构化结果，不返回无法验证的长篇自然语言健康结论。

### Repository 层

- 只负责数据库读写。
- 不写业务决策。
- 不做权限判断的替代实现。
- 不拼接来自 LLM 的自由 SQL 或自由查询条件。

### Agent 层

- Agent Core 独立放在 `backend/app/agent/`。
- Workflow 只负责编排，不创建数据库 session。
- Agent Tools 只能调用 service。
- Harness 负责 runtime、权限、工具门禁、安全、追踪、错误处理。
- LLM 只负责意图识别、结构化草稿、自然语言表达。

## 数据库事务规则

- 数据库 session 必须由基础设施层统一创建和注入。
- Repository 不主动提交事务，除非后续规范明确允许。
- Service 负责定义事务边界或调用统一事务封装。
- 写入健康档案必须能追踪到用户确认、操作者和目标用户。
- 失败时必须 rollback，不允许吞掉写入失败。

## 异步任务规则

- 异步任务放在 `backend/app/jobs/` 或 `backend/app/workers/`。
- 任务可以调用 service，不直接绕过 service 写数据库。
- 定时任务、文档处理、日报生成、通知发送必须保留幂等设计。
- 异步任务失败必须可观测，后续需要接入日志、trace 或审计。
- 不允许异步任务绕过权限或用户确认写入正式健康档案。

## 测试要求

- 文档任务至少检查文件存在、内容可读、路径正确。
- Python 代码任务必须运行相关 `pytest`、lint 或类型检查。
- 前端代码任务必须运行相关 lint、typecheck、test 或 build。
- 数据库迁移任务必须运行 migration 检查。
- Agent 相关任务必须覆盖权限、工具门禁、Safety、未确认不写入等关键场景。
- 测试失败必须先修复再结束；如果由外部环境阻塞，必须说明阻塞原因和未验证风险。

## 医疗安全规则

- 不实现医学诊断。
- 不实现处方建议。
- 不实现药物剂量建议。
- 不输出开药、停药、换药建议。
- 不把系统无记录表述为现实无问题。
- 健康输出必须经过 Safety 边界。

## 任务范围控制

- 每次只实现用户明确要求的任务。
- 不提前实现后续 Phase。
- 不创建当前任务不需要的 API、数据库模型、service、workflow。
- 不改动既有架构文档，除非任务明确要求。
- 不引入与当前任务无关的依赖。
